# Changelog

## [1.11.3] - 2026-05-09 — User-persona/NPC duplicates + delete-does-nothing

### Fixed
- **User personas got duplicated as NPCs, and the Workshop's Delete button silently no-op'd on the persona tile.** Three intertwined bugs:
  - **`removeCharacter` (Send to Workshop) seeded a `knownCharacters` NPC entry unconditionally.** That guard was added so AI-discovered characters with no roster record aren't stranded — but firing it on a portrait-bar click against the user-persona tile created an NPC stub for the persona's name. The persona then appeared in both the Roster Users and Characters tabs and rendered as two PCP tiles. Skip the seed when `userCharacters[name]` already exists.
  - **The v23 settings migration and the per-chat orphan-adopt routine in `persistence.js` promoted any `removedCharacters` orphan into `knownCharacters`, including persona names.** Users on pre-1.9.2 who soft-removed their persona from the PCP had a `removedCharacters` entry with no matching `knownCharacters` record; on next upgrade the migration created the NPC twin even though the same name was a user persona. Both the global migration and the per-chat orphan-adopt now skip names that are already in `userCharacters`.
  - **`buildActiveUserCharacterEntry` ignored `removedCharacters`.** Clicking "Send to Workshop" on the persona tile added the name to `removedCharacters` and saved, but the persona prefix re-rendered next frame regardless — making the soft-remove look like a no-op for personas. Now filters by the active removed list, mirroring how `getCharacterList` filters NPCs.
  - **PCP right-click → Open in Workshop dispatched `{ characterName }` only — no `isUser` flag.** Workshop always opened in NPC mode regardless of which tile was clicked, so deleting from the persona tile wiped the (possibly empty) NPC stores and left `userCharacters[name]` untouched. The persona kept rendering and the user saw "delete did nothing." Right-click handler now reads the tile's `data-user="1"` stamp and forwards `isUser` so the Workshop opens in the correct mode.
  - **`getCharacterList` rendered two tiles for any name that lived in both `userCharacters` and the NPC stores.** The user-prefix tile (with the YOU badge / `data-user="1"`) and the NPC-side tile sat side-by-side, and right-clicking the NPC-side one correctly opened NPC mode because that tile genuinely *was* an NPC entry — fix D had nothing to forward. Now drops the NPC-side render whenever its name matches the active user persona, so there's exactly one tile per name and the right-click route is unambiguous.
  - **`openCharacterWorkshop` infers mode from the data first, dispatch hint second.** If a name only has a `userCharacters` entry, Workshop opens in user mode regardless of where the click came from; if only NPC stores hold it, Workshop opens in NPC mode. The dispatch's `isUser` flag is now a tiebreaker for genuine collisions (both namespaces hold the name) or for brand-new characters being created, instead of the sole source of truth. Closes the case where a stale NPC stub for a persona name forced Workshop into NPC mode.

For users who already have an NPC/persona duplicate created before this release: open the unwanted side via the Roster (Characters tab for the NPC twin, Users tab for a persona twin) and Delete from the Workshop. The other side stays intact.

A deeper refactor — giving every character a stable ID with a user/NPC variant prefix, decoupling identity from display name — is tracked in [#28](https://github.com/DangerDaza/Dooms-Enhancement-Suite/issues/28). The fixes above remove the user-visible symptoms; the ID refactor will eliminate the entire failure class and unlock multiple characters sharing a display name.

### Review-driven follow-ups
- **Case-insensitive name lookups across the persona-collision guards.** The original A and B fixes used exact-key matching (`userCharacters?.[name]`) for the userCharacter check. AI-cased tile names ("yashiro") wouldn't match the canonical persona key ("Yashiro"), so the duplicate-creation guard fell through silently. `removeCharacter` (Send to Workshop) and the v23 / per-chat orphan-adopt paths now compare via `.toLowerCase()`.
- **Workshop mode resolution honors per-chat tracking.** The `npcExists` check read `extensionSettings.knownCharacters` and `extensionSettings.characterColors` directly, both of which are stale when `perChatCharacterTracking: true` (the live data lives on `chat_metadata.dooms_tracker`). A real per-chat NPC sharing a persona's name would be missed and the Workshop would open in user mode incorrectly. Now uses `getActiveKnownCharacters()` / `getActiveCharacterColors()` and matches case-insensitively.
- **Standing ban-list extension prompt refreshed on delete.** `deleteCharacter` (Workshop) and `purgeCharacter` (Roster) stripped `bannedCharacters` lists but never re-emitted the `BAN_SLOT` prompt, so the AI kept being told to exclude a name we'd just deleted until `CHAT_CHANGED` fired. Both paths now call `refreshBanPrompt()` after the strip; `refreshBanPrompt` exported from `characterWorkshop.js` for cross-module use.
- **Right-pinned PCP context menu now clamps to the right viewport edge.** The previous clamp had a floor (`Math.max(0, ...)`) but no ceiling, so a sufficiently wide menu next to a near-rightmost wrapper could overflow.

### Also fixed
- **Right-click context menu opened underneath the PCP when the bar was pinned left or right.** The wrapper's `backdrop-filter: blur(6px)` creates its own stacking context, so a menu rendered as a sibling inside the wrapper's parent (or with too-low a z-index) got buried under the cards. Menu is now re-parented to `<body>` on every show (escapes any ancestor stacking context) and the z-index is bumped from 10000 → 100000 to outrun ST popups. Placement stays cursor-relative — the z-index/re-parent combo is enough to keep the menu on top of the cards without pushing it visually away from where the user clicked.
- **Workshop opened in NPC mode for the YOU-tagged persona tile if `activeUserCharacter` had no matching `userCharacters[name]` entry.** The earlier mode-resolution treated dispatch `isUser=true` as a hint that data inference could override; that override forced NPC mode whenever a persona name lacked a formal Workshop entry but was still the active persona. The YOU badge is now the source of truth: dispatch `isUser=true` opens user mode unconditionally, and data inference only applies when the dispatch flag is absent (Roster Characters tab clicks, programmatic opens).
- **Enforced invariant: only one user tile in the PCP, ever.** `getCharacterList` now suppresses *any* name that exists in `userCharacters` from the NPC tile lists (`presentChars`/`absentChars`), not just the active persona name. A persona that the AI mentions in `characterThoughts`, or a leftover `knownCharacters` stub for a persona name, no longer renders as an NPC tile. The active persona shows once via the YOU-badge prefix tile; non-active personas don't appear at all.

## [1.11.2] - 2026-05-09 — Deleted characters resurrected on reload

### Fixed
- **Deleting a character only made them disappear until the next reload.** The chat-load init in `persistence.js` runs an "orphan-adopt" routine: for every name in `removedCharacters` that has no `knownCharacters` entry, it creates one with a default `👤` emoji (the v23 settings migration does the same thing globally on extension load). This was added in 1.9.2 to recover characters stranded by an old "Send to Workshop" bug, and it's correct for that case — but `purgeCharacter` (Roster) and `deleteCharacter` (Workshop) only wiped `knownCharacters` and left the matching `removedCharacters` entry intact. Any character who had previously been ejected from the scene therefore satisfied the orphan condition (`in removedCharacters && !in knownCharacters`) on next reload, got re-adopted, and reappeared in the Roster with a placeholder portrait. Both delete paths now also strip the name from `removedCharacters` (and from `bannedCharacters`, so a deleted character's standing "do not include" prompt no longer pins) in both the global and chat-scoped stores.

## [1.11.1] - 2026-05-09 — Roster + Workshop deletion fixes

### Fixed
- **Roster delete left the tile in the grid when `perChatCharacterTracking` was on (the default).** `purgeCharacter` in `characterRoster.js` and `deleteCharacter` in `characterWorkshop.js` only wiped the global `extensionSettings.knownCharacters` / `characterColors` maps. With per-chat tracking on, those entries actually live on `chat_metadata.dooms_tracker.knownCharacters` / `characterColors`, which is what `renderGrid` reads from via `getActiveKnownCharacters()` / `getActiveCharacterColors()`. The toast fired, settings saved, the portrait disappeared from the bar — but the next render pulled the chat-scoped entry right back and the tile reappeared. Both functions now also delete from the active maps and call `saveChatData()` when per-chat tracking is on. Workshop and Roster delete paths stay symmetric.
- **Workshop expressions: Upload / Upload ZIP / Delete / Remove All / Open Folder operated on the previously-viewed character.** Two compounding issues. First, the handlers read the character name via `$tab.data('character')`; jQuery's `.data()` caches each `data-*` attribute on first read per element and never refreshes from the DOM, so once the Expressions pane was used for one character, subsequent opens for a different character kept resolving to the first. Second, even after switching reads to `.attr('data-character')`, the attribute itself was only refreshed inside the Expressions tab's nav-button click handler — if the Expressions tab was already active when the user re-opened the Workshop for a different character, the attribute stayed stale. Net effect: the confirmation prompt and toast named the wrong character, and the underlying `/api/sprites/*` calls actually targeted that wrong character — i.e. uploading a ZIP for the open character could overwrite a *different* character's sprite folder, and Remove All could wipe sprites for someone other than who was on screen. Final fix sources `charName` directly from `draft?.name` (the module-scope variable set in `openCharacterWorkshop` and cleared in `closeCharacterWorkshop`, which is also what renders the Workshop title) in all five handlers, so it always matches the visible character.
- **Workshop handlers leaked across extension hot-reload, causing the same wrong-character symptom even after the data-source fix.** When SillyTavern reimports an extension's JS bundle without a full page refresh, the previous module instance's jQuery handlers stay bound to the DOM (`$modal` for `.cw`, `document` for `.cwExpr`) and keep firing alongside the new ones. Each old handler's closure captured the prior module's `draft` variable, so the older copy would resolve to whichever character was active under that earlier module — bug reproduces even when the new code is correct. `bindStaticListeners` and `bindExpressionHandlers` now `.off('.cw')` / `.off('.cwExpr')` before re-binding so only one set of handlers (the current module's) is live.

## [1.11.0] - 2026-05-08 — Pass 2 perf: avatar storage migration

### Changed
- **Cropped portraits now live on disk, not in `extensionSettings`.** Pass 2 of the perf refactor moves base64 PNG data URLs out of the JSON settings file and onto disk under `data/default-user/user/images/des-portraits/<filename>`. A 660×880 PNG is ~250–500 KB of base64; with `*FullRes` siblings that's ~500–900 KB per character. For a 20-character user, settings shrinks by ~10–18 MB, and `saveSettingsDebounced()` no longer re-serializes that mass on every tick.
  - **Zero user disruption.** No folders to move, no portraits to re-apply, no config to change. Migration runs silently in the background on first 1.11.0 boot.
  - **Downgrade-safe.** Migrated entries hold absolute URL strings (e.g. `/user/images/des-portraits/aria-a3f1b2c8.png?t=...`) in the same legacy fields. v1.10.7's `<img>.src = npcAvatars[name]` continues to render fine — same shape as imported `/characters/<file>` URLs.
  - **Failure-safe.** If the upload endpoint is unreachable or rejects a single PNG, that entry stays as a data URL and renders unchanged. Migration retries on next boot. Per-entry try/catch — one bad PNG never aborts the batch. A `__avatarBackupV1` snapshot lives inside settings as belt-and-suspenders against corruption (retired in v1.12).
  - **Cache-busting baked in.** The `?t=<mtime>` query string updates on re-upload, so any mounted `<img>` tags pick up new bytes naturally without DOM walking.
  - Write target: `POST /api/images/upload` with `ch_name='des-portraits'`. Filename format: `<sanitizedName>-<8charHex>.png`; the hex suffix is generated once and persisted in the URL, so re-saving the same character overwrites the same on-disk file.
  - Settings `version` bumps from 23 → 24 only after every entry is URL-shaped, so a partially-completed run resumes on next boot.
  - Touched: `src/utils/avatars.js` (new helpers `uploadPortraitToDisk` / `persistPortrait` / `extractDesPortraitFilename` / `deletePortraitFromDiskByValue`), `src/utils/avatarMigration.js` (new module), `src/core/persistence.js` (v24 migration block), `src/systems/ui/characterWorkshop.js` (commitDraft + delete paths), `src/systems/ui/portraitBar.js` (drop handler + removePortrait), `src/systems/ui/characterRoster.js` (purgeCharacter), `src/systems/features/avatarGenerator.js`, `src/systems/rendering/thoughts.js`, `src/core/state.js` (comment).
- **Bunny Mo Integration is always on, no toggle.** The Enable/Disable switch in the settings panel just gated user discovery — most users never knew the feature existed because they never flipped it. Replaced the toggle row with a how-it-works blurb explaining the three-step flow (run `!fullsheet` → click the scroll icon DES adds to the message → right-click in PCP → Character Sheet). Dropped the on/off badge in the accordion header. The `bunnyMoIntegration` settings field is kept for back-compat (default flipped to `true`), with consumer gates removed in `characterSheet.js`, `portraitBar.js`, and the per-message scroll-icon injection in `index.js`. The "Character Sheet" right-click menu item now always shows.
- **PCP right-click menu trimmed.** Removed three actions from the portrait-card context menu: **Upload Portrait**, **Set Dialogue Color**, **Clear Dialogue Color**. These all live in the Workshop now (Appearance tab), and having them duplicated in two places meant chat-scoped vs global color stores could diverge silently. Workshop is now the single source of truth for portrait + color edits. The orphan color-picker `<input type="color">` and its change handler were removed too. Kept in the menu: Open in Workshop, Cancel Injection, Remove Portrait, Character Sheet, Open Expression Folder, Send to Workshop.

### Fixed
- **Workshop dialog color read used the wrong store with `perChatCharacterTracking` on (the default).** NPC draft hydration in `characterWorkshop.js` read from `extensionSettings.characterColors[name]` directly, but real colors live in `chat_metadata.dooms_tracker.characterColors` — that's where the auto-assigner in `portraitBar.js:440` writes them and where `commitDraft` writes them via `getActiveCharacterColors()`. Symptom: opening the Workshop on a chat-scoped character showed `draft.color = ''` even when the PCP was rendering that character with a color, the preview fell back to the `#e94560` default, and re-opening after manually picking a color showed the read fallback instead of the saved value (looked like Set Dialog Color was a no-op). Fix: read via `getActiveCharacterColors()` so reads and writes go to the same store. User-character draft hydration was already correct.
- **Character Roster missed characters auto-discovered in the current chat.** Same bug pattern as the Workshop color read: `collectCharacterNames` read `extensionSettings.knownCharacters` and `characterColors` directly, but with per-chat tracking on those stores are empty for chat-discovered characters. Symptom: a new character appearing mid-chat showed up in the Present Characters Panel (which builds from `getCharacterList`) but not in the Character Roster grid (which builds from `collectCharacterNames`), making the Workshop unreachable. Fix: read via `getActiveKnownCharacters()` and `getActiveCharacterColors()`.
- **Avatar migration was bumping `settingsVersion` to 2 instead of 24.** The version-bump constant was left at the plan-document's placeholder value (1→2) instead of the actual version chain (23→24). On a successful migration run, `avatarMigration.js` would silently downgrade users from `settingsVersion=23` to `settingsVersion=2`, causing all v3-v23 DES migrations to re-run on the next boot. Most are idempotent (re-applying defaults, no data loss), but v8 in particular re-enables `sceneTracker` widgets that the user may have disabled. Four constants fixed in `src/utils/avatarMigration.js`.
- **New Library `+` button rendered as a filled orange box.** The `.rpg-lb-tab-add` CSS didn't set `background` or `border` so the button inherited a generic `<button>` background (from ST core or browser default) that painted it as a filled orange rectangle, while sibling buttons in the same row had explicit `background: none` and looked correct. Locked `background: transparent; border: none; outline: none; appearance: none;` on both rule occurrences plus the `:hover` overrides. Also adopted the muted `#888` glyph color (was reading `var(--rpg-text)` which renders bright white in the default theme).
- **Chat bubbles collapsed multi-paragraph messages into one giant bubble.** Two compounding issues: (1) the renderer marked any consecutive same-speaker segments as `dooms-bubble-continuation` (compact, no header, no border-top), so a message of N narrator paragraphs visually merged into one bubble with one TTS button. (2) the parser's `getTopLevelBlocks` only walked direct `childNodes` of the container, so a `<p>` containing many `<br>`-separated paragraphs (the common ST output when `simpleLineBreaks: true`) became one block — and presets that wrap content in a styled `<div>` swallowed the inner `<p>` paragraphs entirely. Fixes: tag each segment with its source `_block` index and gate `isContinuation` on both same-speaker AND same-block; rewrote `getTopLevelBlocks` as a recursive walk that recognizes `<p>`, `<br>` (at any nesting depth), `<div>` (recurse into), and `\n` in text as block boundaries. Now produces one bubble per visible paragraph with its own TTS button.

## [1.10.7] - 2026-05-06

### Added
- **Workshop Expressions tab: "Remove All" button.** Bulk-deletes every uploaded expression sprite for the character currently open in the Workshop. Re-fetches the live sprite list before deleting (prevents stale-DOM mismatches), runs deletes in parallel via `Promise.allSettled` so a single failure can't strand the rest, and emits a partial-failure toast naming the success/failure split. Only renders when at least one sprite is uploaded. Shared by user and NPC views since the same renderer drives both.

### Removed
- **TTS Highlight feature.** The Gradient Glow Pill sentence-highlighter for SillyTavern's TTS reads has been removed in full. Performance audit found the feature ran a `setInterval(_pollTtsState, 500)` poll for the lifetime of the page (2 timer wakeups per second, indefinitely) plus a `speechSynthesis.speak` monkey-patch and DOM sentence-wrapping work — all paid even when TTS Highlight was set to `off`. Idle CPU drops noticeably with this gone. The chat-bubble TTS button (the "Read from here" speaker icon on chat bubbles) is unrelated and unchanged. Removed: `src/systems/rendering/ttsHighlight.js`, the TTS Highlight settings accordion from `template.html`, all `dooms-tts-*` CSS rules from `style.css`, and the `ttsHighlightMode` / `ttsHighlightSettings` keys from `state.js`. Stale settings in users' saved data are silently ignored. `manifest.json` description trimmed accordingly.
- **Duplicate Expressions tab on the Character Sheet.** The Workshop's Expressions tab does the same job (and now has Remove All) — keeping a second copy on the Character Sheet just made two surfaces to maintain and confused users about which one was canonical. Removed the tab declaration, content slot, lazy-load wiring, all four sprite handlers (single upload, single delete, ZIP upload, open folder), `renderExpressionsTab()` and the `EXPRESSION_LABELS` constant, the `invalidateSpriteCacheFor()` dynamic-import helper, and now-orphaned `getRequestHeaders` / `getBase64Async` imports from `characterSheet.js` (-244 lines). The Workshop carries its own copies of those helpers so nothing surviving was load-bearing.

### Changed
- **Narrator Mode toggle now has a hint description.** `template.html` row was bare label + switch with no explanation. Now wraps the label in `.rpg-setting-label-group` with a hint: "Infer characters from scene context instead of using fixed character references — useful for narrator-style cards that describe the world rather than play a single role." Matches the layout of every other accordion toggle.

### Fixed
- **Active user character's portrait now updates with expression swaps.** `resolvePortrait` had the user-character avatar shortcut returning before the expression-sync lookup, so user characters got the static persona portrait every render even when the classifier had picked a fresh sprite. The label tooltip updated correctly because that's read separately, but the IMG `src` was always the static avatar — exactly what the "user expressions don't change" reports were describing. Reordered so `getExpressionAwarePortrait` runs first; `resolveFullPortrait` already had this ordering, `resolvePortrait` now matches. NPCs were unaffected (their branch came after the expression check already).
- **User-character expression now refreshes on every AI reply, not just on typed user messages.** `classifyActiveUserExpression` was wired only to `USER_MESSAGE_RENDERED`, so swipes and regenerations of the AI reply left the user's portrait stuck on whatever was last classified when they typed. `onMessageReceived` now also re-classifies the user's last message text alongside the existing AI-side classifier, mirrored across both together and separate/external generation modes. New helper `findLastUserMessageText` walks chat[] backwards skipping synthetic tracker messages from other extensions to source the text.
- **GuidedGenerations interop: synthetic tracker messages no longer corrupt DES state.** GG inserts synthetic messages into `chat[]` with `extra.type ∈ {situationaltracker, stattracker, trackernote}` and `extra.api='manual' / extra.model='tracker system'`, then re-emits `MESSAGE_SENT` and `USER_MESSAGE_RENDERED` for them. DES treated these as real chat turns — `apiClient.js` wrote `extra.dooms_tracker_swipes` onto them, `sillytavern.js`'s `onMessageReceived` ran `parseResponse` on GG's `<details>` HTML in both together AND separate/external mode, `index.js`'s `USER_MESSAGE_RENDERED` + `CHARACTER_MESSAGE_RENDERED` handlers ran chat-bubble styling, reasoning-TTS button injection, and user-character expression classification on them, and `chatBubbles.js` mangled GG's tracker UI. New `src/utils/messageGuards.js:isSyntheticTrackerMessage` predicate gates every touch site, with a defense-in-depth check inside `applyChatBubbles` itself so future callers can't bypass.
- **GuidedGenerations interop: broader inject detection.** `suppression.js` only watched `chat_metadata.script_injects.instruct`. GG actually writes 10 different ids (`instruct, thinking, state, clothes, rules, situational, custom, customAuto, tracker, fun`). Auto-trigger guides bypassed our guided-generation detection entirely, so users on `skipInjectionsForGuided='guided'` got DES tracker injections during every state/clothes guide cycle. Now watches all 10 ids; impersonation pattern matching scans every active guide's content; new `activeInjectIds` field on the return object for diagnostics.
- **System Log: filtered noisy external extension chatter.** GG's `[AUTOTRIGGER-DEBUG]` lines (each with full stack trace), plus QR2 / TunnelVision / STMemoryBooks per-message logs, were rotating real DES errors out of the 500-entry buffer before users could copy them. `systemLog.js` `capture()` now drops these prefixes at capture time. Underlying console still gets them, so devtools is unaffected; only DES's diagnostic bundle is filtered.
- **Skip Injections finally fires for guided swipes (the actual release-blocking bug).** `evaluateSuppression` was reading `context.chatMetadata.script_injects` (camelCase) — that property doesn't exist on `getContext()` on any modern ST build, so the injects map silently collapsed to `{}` and every script_injects-driven detection branch was a no-op. Reproduced cleanly: GG's `guidedSwipe.js` `await`s `/inject id=instruct ephemeral=true` and verifies the entry is in `chat_metadata.script_injects.instruct` before firing `swipe_right()`, yet our diagnostic still printed `allScriptInjectKeys=[]`. Switched to importing `chat_metadata` directly from `script.js` (the live ES-module binding `portraitBar.js` / `nameBan.js` / `characterSheet.js` already use); kept `context.chat_metadata` and `context.chatMetadata` as fallbacks so test harnesses that don't load the real `script.js` still work. After the fix, guided swipes show `activeInjectIds=[instruct]` → `shouldSuppress=true` and DES tracker prompts no longer leak into the guided generation.
- **Skip Injections: late-suppression check after the slash-command queue drains.** Even with the `chat_metadata` fix, a timing race remained: GG's `/inject` is dispatched via `executeSlashCommandsWithOptions` and the slash queue may not have drained when `GENERATION_STARTED` fires. New `onGenerationAfterCommands` handler (registered against `event_types.GENERATION_AFTER_COMMANDS` if the build exposes it) re-runs `evaluateSuppression` with the cached type/data after the queue drains but before the prompt builds, and clears every DES extension prompt slot plus the queued `pendingContextMap` if suppression should now apply. Defensive: gated on `currentSuppressionState` so it no-ops when the early-suppress already cleared things; `lastGenerationArgs` is wiped on `GENERATION_ENDED` / `GENERATION_STOPPED` so stale args can't leak into the next gen. Backward-compatible: emits a console warning and skips registration if the host ST build doesn't expose the event.
- **Skip Injections: native ST Impersonate button now detected via the event type arg.** Regex-only detection (the impersonation patterns in `suppression.js`) misses ST's native Impersonate button on some instruct presets because the prompt language doesn't always match GG-flavored phrasing. `onGenerationStarted` now forwards `type` to `evaluateSuppression`, which short-circuits to `isImpersonationGeneration=true` whenever the event arg is `'impersonate'`. Regex-based detection is kept as a secondary signal for GG impersonation flows that fire `type='normal'` but inject impersonation prompt content via `id=custom` / `customAuto`.
- **Workshop user-only Identity fields were never visible.** `template.html` declared the Pronouns chips and the Linked SillyTavern persona dropdown as `<div class="rpg-field cw-user-only" hidden>`. `style.css` had a hard-override rule (`#character-workshop-popup[data-mode="user"] .cw-user-only[hidden] { display: none !important }`) that kept those elements hidden even when `data-mode="user"` was correctly stamped on the popup. Without the dropdown the user could never set `linkedPersona`, so `mirrorAvatarToPersonaFile()` silently skipped on every save and ST's persona file never got the new bytes. Removed the `hidden` attribute from both blocks AND removed the dead override rule, replacing it with a comment warning future contributors not to re-introduce it. The default `.cw-user-only { display: none !important }` rule still hides the fields in NPC mode, and `[data-mode="user"] .cw-user-only { display: revert }` reveals them in user mode as intended.
- **Workshop portrait → SillyTavern persona file sync (the trigger now actually fires).** Originally added in this release as `mirrorAvatarToPersonaFile()`, but the trigger was gated on `(dirty.avatar || dirty.linkedPersona) && draft.linkedPersona && draft.avatar` — which silently no-op'd when the user opened the Workshop, found the dropdown already showing the right persona from a prior save, and clicked Save without re-touching anything. Confirmed in the field: no toast on save after linking the persona for the first time and uploading nothing new. Mirror is idempotent (writes the same bytes regardless of dirty state), so dropped the dirty gate entirely and just fire on every Save when both fields are populated. When the mirror can't run because something's missing, a new info toast names the missing piece (no linked persona / no portrait uploaded) so users aren't left guessing.
- **Workshop: persona panel `<img>` tags now refresh after mirror without a hard reload.** The original cache-bust used `fetch(url, { cache: 'reload' })`, which only refreshes the browser's HTTP cache for that URL — `<img>` elements already on the page (persona panel preview, message-header user avatar, persona dropdown thumbnails) keep their src and don't repaint. After a successful upload, walk the DOM and append a `?t=<timestamp>` to every `<img>` whose src points at the persona filename so they reload immediately. Adds a green success toast naming the file we wrote so it's obvious the mirror actually fired.
- **Workshop portrait upload now syncs to the linked SillyTavern persona file.** Uploading a new picture for a user character with `linkedPersona` set saved only to `extensionSettings.userCharacters[name].avatar` as a base64 data URL — the persona file on disk was never touched, so the persona panel and `{{user}}` avatars in chat kept showing the original. New `mirrorAvatarToPersonaFile` helper in `characterWorkshop.js` POSTs the same image to `/api/avatars/upload` with `overwrite_name=<linkedPersona>` (matches what ST's persona panel does). Refuses if `linkedPersona` isn't in `power_user.personas` to avoid creating orphan files. Fire-and-forget — DES's local save already succeeded, network failure just toasts a warning.

## [1.10.4] - 2026-05-05

### Fixed

## [1.5.7] - 2026-02-23

### Fixed
- **TTS sentence highlight timer drift** — when `boundary` events from the Web Speech API aren't available, the timer-based fallback was drifting ahead of the actual voice over long messages. Three targeted fixes applied:
  1. **Pause/resume at chunk boundaries** — SillyTavern splits long messages into ~200-char chunks. The timer now pauses when a chunk's `end` event fires and resumes when the next chunk's `start` fires, eliminating the 50–300ms inter-chunk gap that was being counted as speech time (×10 chunks = up to 3 seconds of phantom drift).
  2. **Chunk position resync** — when each new chunk starts, the first significant word in the chunk is used to detect backward drift (timer ran ahead of the engine). If drift is detected, the timer snaps back to the correct sentence.
  3. **WPM calibration** — actual chunk duration (from `start` to `end` events) is measured and used to calibrate the ms-per-word estimate via exponential moving average, adapting to the actual voice speed rather than relying solely on the 310ms/word baseline.
